# Reference designs & training — SafeUpdater · DVBE

These two don't fit any other category. They're not scanners, frameworks, or platforms — they're **artifacts to learn from**. Treat them as documentation that compiles.

| | SafeUpdater (Doyensec) | DVBE (Damn Vulnerable Browser Extension) |
|---|---|---|
| Type | Reference design (defensive) | Training target (offensive learning) |
| What it is | Secure Electron auto-update framework | Intentionally-vulnerable Chrome extension |
| Author | Doyensec | infosecak + Krishna Chaganti |
| Language | JavaScript (Electron / Node.js) | JavaScript (manifest v3 Chrome ext) |
| Repo size | 2.7 MB / 31 files | 644 KB / 12 files |
| Stage | Reference design (production-grade-shaped, but explicitly "not extensively tested") | Vulnerable-by-design teaching example |

The two are at opposite ends of the same axis: **how is software updated / executed in modern desktops, and how does that go wrong?**

---

## SafeUpdater — a deliberately-restrained Electron updater

The context: Electron applications are notorious for unsafe update paths. The two most common patterns are:
1. **electron-updater + GitHub Releases over HTTPS** — better than nothing but no cryptographic verification.
2. **Hand-rolled "fetch URL, replace binary, restart"** — varies wildly in quality.

SafeUpdater is Doyensec's reference design for "what a secure Electron updater looks like." The README is explicit:

> **CAUTION**: This software is provided as-is and has not undergone extensive testing.

That caveat is doing work — Doyensec isn't trying to ship a competing product; they're showing how it *should* be built so other Electron developers can learn from it.

What's actually novel as a reference:

1. **Forked from Signal Desktop's updater logic.** Signal has one of the few production Electron updaters with strong cryptographic guarantees. SafeUpdater extracts that logic as a reusable library. This is the right starting point — Signal's threat model and security posture are documented and audited.
2. **Ed25519 signatures end-to-end.**
   - Long-term keypair generated by `tools/generate.js`.
   - Public key embedded as hex in `config/default.json`.
   - Every release artifact is signed via `tools/sign.js`, producing `.sig` files alongside ZIPs/DMGs/YAMLs.
   - **Signature chain**: independent signatures for ZIP, DMG, YAML metadata, and version manifests. Compromising one artifact doesn't compromise others.
3. **SHA-512 integrity check** on top of signature. Defense in depth — bit-flip detection separate from signature verification.
4. **Certificate pinning** (`certificateAuthority` config option) optionally pins the TLS CA. This prevents trust in compromised root CAs from breaking the update path.
5. **`allowInsecureTLS` is explicit and warned against** — not the default. Doyensec made it possible to use during dev (self-signed cert) but the README screams ⚠️ WARNING about production use.
6. **Path validation** prevents directory traversal in extracted update artifacts.
7. **Secure temp files** — 0o700 permissions (owner-only) for the staging directory.
8. **Verified downgrade support.** Most updaters refuse to downgrade entirely; SafeUpdater allows verified rollback to older versions with the same crypto guarantees as upgrades. Important for revoking a bad release. The user gets a version-selection dialog when `downgradeEnabled: true`. Downgrades are bounded by a `minimumVersion` to prevent rollback to an old vulnerable version.
9. **Update polling with jitter and retry backoff.** Default 30 minutes, randomized; transient errors retried up to a 1-day delay. **Release rollout via hash-based delay** (max 6 hours) — different clients get the update at different times, so a bad release doesn't take everyone down simultaneously.
10. **Atomic operations with rollback.** If the install fails halfway, SafeUpdater rolls back gracefully — no half-applied updates that brick the app.
11. **macOS-only.** Honest scoping. Linux / Windows are future work. Doyensec didn't pretend to cover all platforms.

What's NOT in SafeUpdater:

- No TUF/Notary integration. Just direct signatures.
- No staged rollouts per-channel (beta/stable/canary) — only hash-based delay.
- No update server reference implementation — you bring your own (S3, GitHub Releases, custom).

### Why this matters

SafeUpdater is a **threat-model document that runs**. Doyensec's accompanying blog post lists four unmitigated threat classes in typical Electron updaters: **downgrade attacks, integrity violations, race conditions, untested version deployment**. Each is addressed by a specific architectural choice in the code. Reading SafeUpdater as documentation teaches you the threats first, then shows the mitigations.

For any team shipping a desktop application, **the SafeUpdater README is the right "what's the bar for a secure updater" checklist**, regardless of whether you adopt the code.

## DVBE — Damn Vulnerable Browser Extension

The training-target counterpart. DVBE is to browser extensions what DVWA is to web apps, juice shop is to APIs, and OWASP WebGoat is to JVM web stacks.

It's tiny — 12 files, 644 KB. The point isn't size; the point is what it teaches:

1. **Notes-taking extension as the cover story.** Looks like a normal Chrome extension that lets you write notes. Underneath, it has multiple intentional vulnerability classes you have to discover.
2. **Demonstrates the Chrome extension threat model.** Most developers (and security engineers) have never read [Chrome's extension security docs](https://developer.chrome.com/docs/extensions/mv3/security/). DVBE forces you to confront:
   - `manifest.json` permission requests and what each permission grants
   - Background script vs. content script isolation
   - Web-accessible resources
   - Cross-Origin Read Blocking (CORB)
   - Content Security Policy (CSP) interactions
   - Storage access (sync vs. local)
   - Message passing between contexts
3. **Vulnerable JavaScript patterns specific to extensions.** Things like:
   - `innerHTML` from untrusted message contents → DOM XSS in the popup
   - Missing CSP allowing inline scripts in popup.html
   - Over-permissive `host_permissions`
   - Improper use of `chrome.tabs.executeScript` or `chrome.scripting.executeScript`
   - Sensitive data stored unencrypted in `chrome.storage.local`
4. **Loaded via Chrome Developer Mode.** No build step. Drop into `chrome://extensions` → "Load Unpacked." Lowers the friction to zero.
5. **BlackHat Arsenal twice** — Asia 2024 and Europe 2024. Used in workshop sessions. The teaching loop is field-tested.
6. **Tiny enough to read end-to-end.** Even non-developers can study the entire codebase in 20 minutes. Compare to DVWA (4000+ files of PHP) or WebGoat (thousands of files of Java) — DVBE is *the* extension-security teaching artifact.

What's NOT in DVBE:

- No write-up of the intentional vulnerabilities (the README says it will be released later).
- No automated grading or scoring.
- No multi-stage attack scenarios (it's a single static target).
- Single browser (Chrome). Firefox / Edge extension specifics aren't covered.

### Why this matters

Browser extension security is **massively under-trained** in the security industry. Every developer learns about web XSS; almost none learn extension-specific patterns like content-script-to-page-context communication risks. Chrome Web Store has 175,000+ extensions; the attack surface is enormous and rarely audited.

DVBE makes "learn extension security" a 1-hour exercise instead of a 3-month research project. Pair it with the [`Adversis/sketchy`](https://github.com/Adversis/sketchy) cross-platform behavioral scanner (mentioned in the conference notes) and you have a defender training loop: write extension → audit extension → break extension → fix extension.

## How to use these in a security program

| Need | Use |
|---|---|
| Ship an Electron app with a secure updater | Read SafeUpdater's README + threat model; either adopt the code or use it as a checklist when building your own |
| Train developers on browser extension security | DVBE as the lab; pair with the OWASP Chrome Extension Security Guide |
| Train AppSec engineers to audit extensions | DVBE again, paired with `chrome://extensions` developer mode and a clean Chrome profile |
| Audit a third-party Electron app | Use SafeUpdater as the reference; check whether the app's updater matches (Ed25519 sig? SHA-512? cert pinning? rollback?) |

## Strategic read

These two are the "training and reference design" tier of the AppSec submodule set. Less immediately deployable than the other tools, but **disproportionately useful for upskilling** — both for developers and for security engineers. If you have a learning budget at all, these are the highest-ROI training material in the 17 because:

- DVBE teaches an under-covered domain (browser extensions) with near-zero setup.
- SafeUpdater teaches secure desktop-update architecture with a complete working example.

Neither has direct competitors of the same quality in 2026.
