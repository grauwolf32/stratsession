## 1. Architecture overview

The CLI entrypoint is a 916-line `main.go` that does flag parsing (`cmd/nuclei/main.go:251`), constructs `*types.Options`, and hands off to `runner.New(options)` (`cmd/nuclei/main.go:177`) before calling `nucleiRunner.RunEnumeration()` (`cmd/nuclei/main.go:237`). The runner (`internal/runner/runner.go:509`) is the orchestrator: it builds a single `protocols.ExecutorOptions` (`runner.go:569`) shared by every protocol; constructs the scan engine via `core.New(r.options)` (`runner.go:632`); creates a `loader.Store` from the catalog (`runner.go:647`); calls `store.Load()` (`runner.go:684`) to parse and compile every YAML template; and finally invokes `engine.ExecuteScanWithOpts(ctx, finalTemplates, inputProvider, …)` (`internal/runner/runner.go:864`, defined in `pkg/core/execute_options.go:34`).

The engine itself (`pkg/core/engine.go:18`) is deliberately thin — only 67 lines — and exists mostly to hold a `WorkPool`. Real execution lives in `pkg/core/executors.go`: `executeTemplateWithTargets` (`executors.go:50`) is a bounded-pool fan-out that iterates the `InputProvider` and dispatches each `(template, target)` pair through `executeTemplateOnInput` (`executors.go:247`), which finally calls `template.Executer.Execute(scanCtx)` or routes to `executeWorkflow` for workflow templates. The scan loop is therefore: load templates → enumerate inputs → for each (template, input) pair invoke `Executer.Execute`/`ExecuteWithResults` → matchers/extractors fire → results go either to a `Callback` or to `executerOpts.Output.Write`.

## 2. Module / package layout

`pkg/templates/` owns the on-disk YAML model (`templates.go`, `compile.go`, `parser.go`, `cluster.go`). `pkg/protocols/` contains one subpackage per wire protocol (`http`, `dns`, `network`, `ssl`, `headless`, `websocket`, `whois`, `code`, `javascript`, `file`, `offlinehttp`) plus `protocols.go` which defines the two core interfaces, `Executer` (`protocols.go:49`) and `Request` (`protocols.go:321`). `pkg/operators/` holds the shared matcher/extractor engine: `matchers/`, `extractors/`, `common/dsl/`, and the top-level `operators.go:232` `Execute()` driver. `pkg/tmplexec/` wraps requests into one of three engines — `generic` (single request), `multiproto` (multi-protocol sequential), `flow` (JS-driven) — selected in `tmplexec/exec.go:41`. `pkg/core/` is the work-pool scheduler; `pkg/output/` is the writer family.

## 3. YAML template engine

The `Template` struct (`pkg/templates/templates.go:35`) is the only place every protocol is enumerated: it carries 11 typed slices — `RequestsHTTP []*http.Request` (line 71/78, where `requests:` and `http:` are accepted as aliases), `RequestsDNS`, `RequestsFile`, `RequestsNetwork`/`RequestsWithTCP`, `RequestsHeadless`, `RequestsSSL`, `RequestsWebsocket`, `RequestsWHOIS`, `RequestsCode`, `RequestsJavascript`, plus an embedded `workflows.Workflow` and a top-level `Flow string` (`templates.go:65`) for the JS flow language. Parsing happens in `parseTemplateNoVerify` (`pkg/templates/compile.go:492`) using `yaml.Unmarshal` (line 505); JSON is also supported (`compile.go:500`). Required-field validation (`ID`, `info.name`, `info.author`, severity) is enforced inline (`compile.go:513-535`). The actual protocol dispatch — choosing what the template "is" — is `Template.Type()` (`templates.go:181`), a cascade of `HasXRequest()` checks producing one of the 12 `ProtocolType` constants from `pkg/templates/types/types.go:21-43`.

Compilation continues in `Template.compileProtocolRequests` (`compile.go:302`): if multiple protocol sections are present, `RequestsQueue` (preserved during unmarshal) is used as-is and `options.IsMultiProtocol = true` is set (line 320); otherwise the single non-empty slice is converted to `[]protocols.Request` via reflection in `convertRequestToProtocolsRequest` (`compile.go:362`). The final step is `tmplexec.NewTemplateExecuter(requests, options)` (`compile.go:355`), and the parser then calls `template.Executer.Compile()` (`compile.go:578`) which fans out to each request's `Compile(options)`. Parsed templates are memoised in two LRU-style caches (`Parser.parsedTemplatesCache` and `compiledTemplatesCache`, `pkg/templates/parser.go:25-35`); the cached re-use path in `Parse` (`compile.go:170-235`) deep-copies the template and re-applies options rather than re-parsing the YAML.

## 4. Protocol implementations

**HTTP** (`pkg/protocols/http/request.go`, 1318 lines). `http.Request` (`pkg/protocols/http/http.go:36`) embeds `operators.Operators` via `yaml:",inline"` (line 38), so matchers/extractors are not registered — they are physically part of the same YAML mapping. `Request.ExecuteWithResults` (`request.go:506`) is the dispatcher: it routes to `executeTurboHTTP` (pipelining, `request.go:509`), `executeRaceRequest` (race condition, line 513), `executeFuzzingRule` (DAST, line 518), or `executeParallelHTTP` (line 523), and otherwise enters the main payload loop driven by `request.newGenerator(false)` (line 526). For each iteration, `generator.Make` builds the wire `*http.Request` (line 542), `executeRequest` (line 569) actually fires it through the pooled HTTP client (`httpclientpool/`), and the response is wrapped into an `output.InternalWrappedEvent` whose `OperatorsResult` is filled by the embedded matchers via `eventcreator`.

**Headless** (`pkg/protocols/headless/`). `headless.Request` (`headless.go:17`) also embeds `operators.Operators` (line 52) and stores a `Steps []*engine.Action` (line 37) — a declarative DSL of browser actions (navigate, click, waitload, extract). `Request.ExecuteWithResults` (`request.go:40`) iterates a payload generator, and for each value calls `executeRequestWithPayloads` (`request.go:114`) which spawns a new browser instance via `request.options.Browser.NewInstance()` (line 115) and runs the action steps with `instance.Run(input, request.Steps, payloads, options)` (line 142). The browser engine (`pkg/protocols/headless/engine/engine.go:35`) launches Chrome through `go-rod` (`launcher.New()` line 39, `rod.New().ControlURL(launcherURL)` line 101) with `Leakless(false)`, incognito, `ignore-certificate-errors`, optional `NoSandbox`, optional `UseInstalledChrome` (musl/alpine fallback at line 71), and an optional CDP endpoint for remote browsers (line 41/98). After execution the page HTML is harvested (`request.go:178`), turned into a DSL map by `responseToDSLMap` (line 189), and fed to the same `operators.Execute` pipeline as HTTP — meaning matchers/extractors are shared code across all protocols.

## 5. Matchers + extractors

Seven matcher types live in `pkg/operators/matchers/matchers_types.go:15-31`: `WordsMatcher`, `RegexMatcher`, `BinaryMatcher`, `StatusMatcher`, `SizeMatcher`, `DSLMatcher`, `XPathMatcher`. Each has a tiny dedicated method on `Matcher`: `MatchStatusCode` (`match.go:23`) iterates `matcher.Status` ints; `MatchSize` (line 39) compares lengths; `MatchWords` (line 55) does `strings.Contains` per word with AND/OR short-circuiting and pre-evaluates DSL expressions inside the word via `expressions.Evaluate` (line 68); `MatchRegex` (line 105) uses pre-compiled `*regexp.Regexp` plus a literal-prefix `strings.Contains` short-circuit (line 112) to skip patterns whose anchor can't possibly match; `MatchBinary` (line 159) does the same against hex-decoded payloads in `matcher.binaryDecoded`. Extractors (`pkg/operators/extractors/extract.go`) come in four flavours: `ExtractRegex` (line 15) with named-group selection via `RegexGroup`, `ExtractKval` (line 47) which pulls map keys with optional case-insensitive folding, `ExtractXPath` (line 74) dispatching to `ExtractXML` or `ExtractHTML` (line 82) via `antchfx/htmlquery`/`xmlquery`, plus a JSON extractor. The orchestrator is `Operators.Execute` (`operators/operators.go:232`): extractors run first so their results feed `data` for matchers (line 273); extractor results with `Internal: true` become `DynamicValues` (line 257) rather than user-visible output, which is how Nuclei chains one request's extract into the next request's body via `{{name}}` interpolation.

## 6. DSL (`pkg/operators/common/dsl/`)

Nuclei does not implement its own expression language. `pkg/operators/common/dsl/dsl.go` is a 141-line shim that imports the external `github.com/projectdiscovery/dsl` library and exposes its function map via `HelperFunctions = dsl.HelperFunctions()` (line 122). That upstream library wraps `github.com/Knetic/govaluate`, and Nuclei registers exactly two custom functions on top of it: `resolve(host, type)` (lines 25-101, doing live DNS lookups against `dnsclientpool`) and `getNetworkPort(port, defaultPort)` (line 102) which substitutes a default for well-known ports. Evaluation happens in `Matcher.MatchDSL` (`pkg/operators/matchers/match.go:190`): pre-compiled `govaluate.EvaluableExpression`s are walked, any unresolved `{{var}}` placeholders are pre-resolved via `expressions.Evaluate` (line 198), then `expression.Evaluate(data)` (line 210) returns the boolean result. Variable interpolation throughout the runtime — `{{BaseURL}}`, `{{Hostname}}`, `{{randstr}}`, helper functions like `base64(...)` — goes through `pkg/protocols/common/expressions/expressions.go:17` which constructs the same `govaluate.NewEvaluableExpressionWithFunctions` using `dsl.HelperFunctions`. So matchers, extractors, request templating, and the `-template-condition` filter all share one expression engine.

## 7. Workflow / flow templates

Two distinct "chaining" mechanisms. **Workflows** are pre-`flow` declarative chaining in `pkg/workflows/workflows.go`: `Workflow.Workflows []*WorkflowTemplate` (line 16), each `WorkflowTemplate` (line 22) carrying a sub-template path, tag selectors, named `Matchers` (line 36) and recursive `Subtemplates` (line 39). Execution lives in `pkg/core/workflow_execute.go:19` (`executeWorkflow`), dispatched from `Engine.executeTemplateOnInput` when `template.Type() == WorkflowProtocol` (`pkg/core/executors.go:253`). The matcher (`workflows.Matcher.Match`, `workflows.go:101`) inspects the previous result's `Matches`/`Extracts` maps to decide whether to descend into `Subtemplates`.

**Flow templates** are the newer JavaScript composition layer in `pkg/tmplexec/flow/flow_executor.go`. When a template declares `flow:` (e.g. `for region in regions { http(0); ssl(1) }`), `tmplexec.NewTemplateExecuter` (`pkg/tmplexec/exec.go:43`) compiles the snippet into a `goja.Program` (`exec.go:47`) instead of constructing a generic engine. At execution time, `FlowExecutor.ExecuteWithResults` (`flow_executor.go:181`) pulls a `*goja.Runtime` from a sized pool (`flow/vm.go:33`, sized by `opts.JsConcurrency`, default 120), then `Compile` (line 110) registers one JS function per protocol present in the template (`http`, `dns`, `ssl`, …) by iterating `allProtocols` and binding each name to `requestExecutor` (line 174). The user's JS gets builtins `set(key, value)` and the `template` global injected (lines 229, 248), so scripts can write `template.found = true; if (http("vuln") && dns("oast")) ssl("verify")`. The runtime is returned to the pool only if `CanRunAsIIFE` (`flow_executor.go:202`) — i.e. it didn't pollute the global scope.

## 8. AI template generation (`-ai` flag)

Wiring is shallow. The flag is registered as `-ai` in `cmd/nuclei/main.go:291` (binding to `options.AITemplatePrompt`) and gated on PDCP credentials in `cmd/nuclei/main.go:570`. The integration point is `pkg/catalog/loader/loader.go:227-233`: during store construction, if a prompt is set, `getAIGeneratedTemplates(cfg.AITemplatePrompt, cfg.ExecutorOptions.Options)` is called and the returned file paths are appended to `store.finalTemplates`, after which the normal template-loading pipeline takes over. The function itself is `pkg/catalog/loader/ai_loader.go:33`: it length-validates (5-3000 chars), POSTs `{"prompt": …}` JSON to the hardcoded endpoint `https://api.projectdiscovery.io/v1/template/ai` (`ai_loader.go:21`) with the PDCP API key header (line 113), decodes an `AITemplateResponse` (line 24, fields `Completion`, `TemplateID`, `CanRun`, `Comment`, `Name`, `Message`), writes the YAML to `<templates-dir>/pdcp/<template_id>.yaml` (line 53), and returns that path. There is no local model, no prompt template, no validation that the returned YAML parses — it relies on the catalog loader's normal parse-and-compile pipeline downstream to reject malformed output.

## 9. What's genuinely novel at the code level

The most distinctive design decision is that **`operators.Operators` is embedded inline in every protocol Request struct via `yaml:",inline"`** (`http.go:38`, `headless/headless.go:52`, etc.) rather than registered through an interface. The same `operators.Execute` driver (`operators.go:232`) thus runs on HTTP responses, DNS records, TLS certificates, headless DOM, and JS return values — meaning a new protocol only needs to implement `Match`/`Extract` over a `map[string]interface{}` to inherit the entire matcher/extractor/DSL system for free. The second novel piece is the **goja-based flow engine** (`pkg/tmplexec/flow/`): rather than build a YAML DSL for control flow, Nuclei reuses the YAML for declaration and embeds an actual JS runtime with sized pooling (`flow/vm.go:30`), exposing each protocol as a JS-callable function, which is what makes constructs like `if http(0) { for (let cve of template["cves"]) http(cve) }` work without inventing new syntax. Third, **template clustering** (`pkg/templates/cluster.go:46`) hashes "clusterable" single-request HTTP/DNS/TLS templates by their request signature and merges identical wire requests across multiple templates into one execution — a substantial performance win on the 10k+ nuclei-templates corpus that is invisible at the YAML level.
