# Little Snitch for Linux — code-level analysis

Submodule path: `sources/defense/littlesnitch-linux/`. Upstream: Objective Development Software GmbH. All paths below are relative to the submodule root.

## Repository scope

This repository is **only the OSS slice** of the shipping product. `README.md:14-22` is explicit: the public tree contains the `ebpf` crate, the `common` crate of shared kernel/user types, a `demo-runner` user-space loader, and a `webroot/` JS front-end. The production user-space daemon (rule engine UI server, persistence, auth, agent UI) is **proprietary and absent** from this repo. `webroot/README.md` even names the missing types file (`daemon/src/client/ui_types.rs`) as the "Contract Source of Truth" — that crate is not here. Practically, the OSS tree is enough to (a) study the in-kernel firewall design and (b) run a stand-alone demo with two text blocklists, but not enough to reproduce the full Little Snitch UX.

## Cargo workspace layout

`Cargo.toml:1-26` declares a 3-member workspace (`demo-runner`, `common`, `ebpf`) with `resolver = "2"`, `edition = "2024"`, and a workspace `license = "MIT OR Apache-2.0"`. Note that every source file is SPDX-tagged `GPL-2.0` and `LICENSE-GPL2.txt` is the only license text shipped (`README.md:50-55`); the workspace `license` field appears stale relative to the actual licensing intent. Workspace pins every `aya*` crate to a specific git rev (`4fbce44b6a49…`, `Cargo.toml:17-21`) "until a newer one is proven to work" — eBPF tooling churn is the stated reason.

- `common/` (`common/Cargo.toml`): `no_std` library that compiles for both the `bpf` target and host; gated `user` feature pulls in `aya` for the `Pod` marker (`common/src/lib.rs:124-129`).
- `ebpf/` (`ebpf/Cargo.toml:9-23`): kernel-side crate; depends on `common`, `aya-ebpf`, `aya-log-ebpf`, `network-types`; produces binary `linux-snitch-ebpf`. Build-deps include `bindgen`, `cc`, and `which` for locating `bpf-linker`.
- `demo-runner/` (`demo-runner/Cargo.toml`): `tokio` async user-space binary depending on `aya`, `aya-log`, `aya-build`, `clap`, and the `ebpf` crate (declared as a regular dep solely for cache invalidation per the comment at `demo-runner/Cargo.toml:28-40`).

## eBPF program design

Confirmed: aya is the framework. `ebpf/src/main.rs:47-60` imports `aya_ebpf::{macros::{cgroup_skb, cgroup_sock, cgroup_sock_addr, fentry, fexit, kprobe, kretprobe, tracepoint}, programs::…}`. The kernel-side binary attaches **eleven programs across five hook families**:

| Hook | Function | Source |
|---|---|---|
| `cgroup/sock_create` | track socket open, stash owning process by socket cookie | `ebpf/src/main.rs:76-81` |
| `cgroup/sock_release` | clean up on close | `ebpf/src/main.rs:83-88` |
| `cgroup/connect4`,`connect6`,`sendmsg4`,`sendmsg6` | pre-connect verdict point (drop = `SK_DROP`) | `ebpf/src/main.rs:90-108` |
| `cgroup/skb` ingress+egress | per-packet accounting and packet-level enforcement | `ebpf/src/main.rs:276-284, 346-364` |
| `fentry/fexit bprm_execve`, `security_bprm_creds_for_exec`, `sched_mm_cid_after_execve` (+`kprobe`/`kretprobe` fallbacks) | observe exec to update the responsible binary for a PID | `ebpf/src/main.rs:114-203` |
| `tracepoint sched/sched_process_{exec,fork,exit}` | PID lifecycle bookkeeping | `ebpf/src/main.rs:205-252` |

Process identification is two-layered. (1) `bpf_get_socket_cookie()` (`ebpf/src/main.rs:78, 85, 289`) keys a `HashMap<u64, SocketProperties>` named `SOCKET_PROPERTIES` (`ebpf/src/socket_properties.rs:8-9`) that records the owning `ProcessPair` at `sock_create` time. (2) The "owner" is computed by walking task_struct/dentry chains via a CO-RE C helper module (`ebpf/co-re/co-re.c`, `co-re.h`) whose Rust bindings are generated by `bindgen` in `ebpf/build.rs:48-61` and re-included as `co_re` in `ebpf/src/main.rs:8-15`. This is how the firewall associates a connection with an executable's *real* inode-path rather than a `/proc`-style string. The comment at `ebpf/src/main.rs:116-135` explains why `bprm_execve` is the chosen interception point — need the file open so a `struct dentry` is available, and need an inner-static function to avoid inlining.

The eBPF-to-userspace channel is an `aya_ebpf::maps::RingBuf` called `EVENT_QUEUE` sized at 1 MiB (`ebpf/src/event_queue.rs:7-19`), used via a closure-style `enqueue_event(|event| { … ; true })` pattern that reserves, fills, and either submits or discards. Verdicts are computed **in-kernel** by `KernelFilterModel::shared().evaluate_network_filter(…)` (`ebpf/src/main.rs:315-326`) consulting blocklist/rule pages laid out in `common/src/network_filter/{blocklist_page.rs, rule_page.rs}`. Enforcement returns `SK_DROP` or `SK_PASS` (`ebpf/src/main.rs:337-340`). The comment at `ebpf/src/main.rs:355-359` documents a verifier workaround: matching all `Verdict` variants would let the compiler emit a raw enum value, which the verifier rejects because it sees the full `u8` range.

Userspace receives a fully-formed `common::event::Event` (process pair, IP, port, protocol, verdict reason, byte counters, change-bitset) — see the merge-bit constants `CONNECT/DISCONNECT/BLOCKED/EXECUTABLE_UPDATED/REMOTE_NAME_UPDATED` at `common/src/event.rs:8-12`.

## Common crate

Two key shared structs:

- `common/src/event.rs:21-28` — `ConnectionIdentifier { process_pair, remote_address, remote_name: StringId, is_inbound, protocol: u8, port: u16 }`. `#[repr(C)]` so it has identical layout on both sides of the verifier/user boundary.
- `common/src/flow_types.rs:23-30` — `FlowIdentifier { protocol: u32, local_address, remote_address, local_port: u16, remote_port: u16 }`. The protocol is widened to `u32` "to avoid padding for byte-wise equality" — i.e. the struct doubles as a hash-map key in `ACTIVE_FLOWS`.

Verdicts are `Verdict::{Allow,Deny}` (`flow_types.rs:80-85`) and `VerdictReason` (`flow_types.rs:87-98`) carries provenance — which blocklist/rule fired. The `user` cargo feature gates `aya::Pod` impls (`common/src/lib.rs:124-129`) so the host can read these structures out of `aya::maps::HashMap` directly.

## Demo runner

Entry point is `demo-runner/src/main.rs`. Sequence:

1. `Opt::parse()` takes `--cgroup-path` (default `/sys/fs/cgroup`, `main.rs:36-37`).
2. `setrlimit(RLIMIT_MEMLOCK, INFINITY)` for older-kernel BPF map accounting (`main.rs:50-57`).
3. `DemoEbpfProxy::new()` does `aya::Ebpf::load(include_bytes_aligned!(.../linux-snitch-ebpf))` (`demo_ebpf_proxy.rs:43-50`) and `take_map()`s eight named maps: `ACTIVE_FLOWS`, `SOCKET_PROPERTIES`, `EVENT_QUEUE` (RingBuf), `DNS_QUERIES`, `DNS_CNAMES`, `DNS_IPV4ADDR`, `DNS_IPV6ADDR` (`demo_ebpf_proxy.rs:52-75`), plus filter maps in `demo_filter_maps.rs:32-60` (`MATCH_TABLE_METAINFO`, `NAME_BLOCKLIST`, `IPV4_BLOCKLIST`, `IPV6_BLOCKLIST`, `NAME_RULES`, `IPV4_RULES`, `IPV6_RULES`, `ANY_ENDPOINT_RULES`, `EXE_PATTERNS`).
4. `node_manager.add_node_features(NodeFeatures::APP_MANAGER, &["/usr/lib/systemd/systemd", "/usr/bin/gnome-shell"])` and walks `/etc/shells` to mark shells as `NON_PARENT` (`main.rs:63-79`), so the process-attribution chain skips them when looking for the "responsible" binary.
5. **Blocklist load + map write**: `filter_engine.load_blocklists(&[(Path("blocked_hosts.txt"), false), (Path("blocked_domains.txt"), true)])` (`main.rs:82-89`). Implementation in `demo_blocklist_load_from_file.rs:9-43` parses, in-memory de-dups, merges host/domain entries; then `demo_blocklist.rs:14-46` sorts entries reverse-byte (with `.` → 0 normalization for proper binary-search ordering), packs them into fixed-size blocklist pages, and `self.name_blocklist.set(index, page, 0)` writes them into the `NAME_BLOCKLIST` `aya::maps::Array`.
6. `attach(cgroup_path)` opens the cgroup directory and calls `program.load(); program.attach(&cgroup, …)` for each of the eleven programs (`demo_ebpf_proxy.rs:116-211`). The exec-tracking probes also require `Btf::from_sys_fs()` (`demo_ebpf_proxy.rs:161`).
7. Main loop `select(2)`s on the ring-buffer FD, draining events and calling `garbage_collect_flows()` every 2 s (`main.rs:104-126`). SIGQUIT triggers a debug dump (`main.rs:99-101, 129-133`).

## Web UI (`webroot/`)

Plain JavaScript, no framework. `webroot/index.html` ships a static SVG-symbol shell and loads `app.js`, `connections.js`, `blocklists.js`, `rules.js`, `traffic.js` (with bundled `uPlot.iife.min.js`), `datetime.js`, `localization.js`, `selection.js`, `sw.js`. The shell at `webroot/index.html:15-30` reads `localStorage.theme` for dark mode.

Backend transport is **WebSocket only** — `webroot/app.js:53-60` opens `ws(s)://<host>/stream` (falling back to `ws://127.0.0.1:3031/stream` for `file://` debugging) and `webroot/app.js:67-73` parses JSON message arrays. There is no REST surface in this tree. The action vocabulary is `setSection`, `setFilterDisabled`, `setBlocklistEntryDisabled`, `addRule`, `deleteRules`, etc.; the reverse vocabulary is `insertConnectionRows`, `setBlocklists`, `trafficEvents`, etc. — all enumerated in `webroot/README.md`. The PWA service worker `webroot/sw.js:1-9` is a stub: install/activate only, no fetch handler. **The HTTP server that exposes `/stream`, serves the static files, and translates messages to/from the eBPF maps lives in the proprietary `daemon` crate**, which is not in this repo (`webroot/README.md` references `daemon/src/client/ui_types.rs` as the contract source of truth).

## Build tooling

- `ebpf/build.rs:19-46` requires `bpf-linker` on `$PATH` (located via the `which` crate), invokes `clang --target=bpf -emit-llvm` on `co-re/co-re.c`, then passes the resulting bitcode to `rustc` via `cargo::rustc-link-arg`. It also runs `bindgen` to materialize the kernel struct field-offset accessors used by the eBPF Rust code.
- `demo-runner/build.rs` uses `aya_build::build_ebpf(…, Toolchain::default())` (line 30) to compile the `ebpf` crate for the `bpf` target as a build artifact. `with-inline-assembler` is enabled (`demo-runner/build.rs:27`), which is what activates the inline-asm `touch_*` no-ops in `common/src/lib.rs:116-122` and the BPF `#[panic_handler]` in `ebpf/src/main.rs:368-372` — those `touch_*` ops prevent LLVM from optimizing out range checks that the verifier requires.
- Release profile override `[profile.release.package.ebpf] debug = 2; codegen-units = 1` (`Cargo.toml:23-25`) keeps DWARF in the BPF object (needed for BTF) and forces a single codegen unit so the verifier sees the canonical IR.
- **No `rust-toolchain.toml` is committed** (only `rustfmt.toml` exists). The README instead documents in prose (`README.md:27-35`) that the project needs nightly + `rust-src` (for `-Z build-std` to compile `core` for the bpf target) and is **known to pass the verifier only on Linux kernels 6.12–7.0 with Rust 1.93.0-nightly (2d4fa1395 2025-11-12)**. Pinning is via prose, not via tooling — an explicit caveat that this is a fragile build matrix.

## License + ABI implications

The eBPF program declares `static LICENSE: [u8; 4] = *b"GPL\0";` in `link_section = "license"` at `ebpf/src/main.rs:374-376`. Every source file (Rust, C, HTML, JS) is `SPDX-License-Identifier: GPL-2.0`, and the only license text is `LICENSE-GPL2.txt`.

This is not an aesthetic choice. The Linux kernel exposes a large subset of BPF helpers (and verifier behaviors) **only to programs declared GPL-compatible** via the `BPF_PROG_LICENSE` check — helpers such as `bpf_probe_read_kernel`, `bpf_get_stack`, `bpf_skb_load_bytes_relative`, and the `fentry`/`fexit` tracing infrastructure all gate on the GPL marker. This program uses `bpf_get_socket_cookie` (`ebpf/src/main.rs:78`) and the full `fentry`/`fexit`/`kprobe` tracing surface, so the `"GPL\0"` declaration is an ABI requirement, not just a license preference. Distributing this code under any non-GPL-compatible license would result in the kernel rejecting program load.

## What's genuinely novel at the code level

The aya + `bpf-linker` + Rust-2024-edition stack with a `common` crate that compiles unchanged to both `target_arch = "bpf"` and the host (gated by a `user` cargo feature for `aya::Pod` impls) is a clean reference for "modern eBPF in Rust" — it avoids the dual-codegen problem that plagues C-based projects relying on `vmlinux.h`/libbpf-skeleton. The two-step process-identification design (socket-cookie hashmap populated at `cgroup/sock_create`, executable identity sourced from `bprm_execve` via a CO-RE C bitcode helper providing real `dentry`/`vfsmount` paths instead of `/proc/<pid>/exe`) is reusable far beyond per-app firewalling — it's a general template for any per-process eBPF policy engine that needs symlink-resistant binary identity. Finally, the `touch_*` inline-asm no-ops (`common/src/lib.rs:116-122`) and the `Verdict` match-exhaustiveness comment (`ebpf/src/main.rs:355-359`) are concrete artifacts of fighting the verifier from Rust, useful as recipes for other projects.
